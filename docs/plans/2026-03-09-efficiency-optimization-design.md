# 小宇宙播客总结工具效率优化设计

**日期：** 2026-03-09
**状态：** 已批准

## 背景

用户希望优化播客总结工具的效率，主要关注：
- 处理速度
- 跳过重复处理
- 资源占用
- 用户体验

**使用场景：**
- 每次处理 1-3 个播客
- 很少重复处理同一播客
- Whisper 转写是主要瓶颈
- M4 Mac 环境
- 保持现有依赖

## 设计方案

### 1. Whisper 分段策略优化

**改动：**
- 分段时长从 300 秒改为 600 秒
- 减少切分次数，降低开销
- 保持 `tiny` 模型作为默认（速度优先）

**代码位置：** `main.py:627-688`

### 2. 进度显示增强

**改动：**
- 分阶段显示（下载 → 转写 → 总结）
- 显示已用时间
- 多播客时显示总进度

**显示格式：**
```
=== 播客 2/3: 《xxxxx》 ===
[1/3] 下载音频... 45.2 MB / 60.0 MB (75%)
[2/3] Whisper 转写... 分段 3/5 (60%) | 已用 2:30
[3/3] DeepSeek 总结中...
```

**代码位置：** `main.py:467-479`, `main.py:1004-1043`

### 3. 错误恢复机制

**改动：**
- DeepSeek API 自动重试 3 次（间隔 5/10/20 秒）
- 超时时间从 30 秒改为 120 秒
- 音频下载失败重试 2 次
- 单个播客失败不影响其他，最后汇总报告
- 错误信息更友好

**代码位置：** `main.py:252-270`, `main.py:531-569`, `main.py:1004-1060`

### 4. 跳过已处理的播客

**改动：**
- 检测输出文件是否已存在，存在则跳过
- 基于 episode_id 匹配现有文件
- 添加 `--force` 参数强制重新处理

**代码位置：** `main.py:878-944`, `main.py:1004`

## 不包含的改动

- 不更换 Whisper 模型（保持 tiny）
- 不添加新依赖
- 不实现并行处理（使用场景为 1-3 个播客）
- 不实现复杂的缓存系统

## 实现顺序

1. 进度显示增强（改善体验，便于后续调试）
2. 错误恢复机制（提高稳定性）
3. 跳过已处理检测（避免重复工作）
4. Whisper 分段优化（小幅提升）
